# Риск, план, код

## `risk`

Классификация риска `low|medium|high|blocked` по полям задачи, областям, планируемым путям и
словам-триггерам (EN + RU). `blocked` → агент не кодит; `high` → нужны Red Team и
human pre-approval. Лимиты числа файлов повышают уровень. Уровень риска **пересчитывается
сервером**, у модели его не берут; `risk_hint` постановщика только повышает.

## `context_engine`

Песочница над распакованным репозиторием: `list_dir`, `grep`, `read_file`. Все пути
нормализуются и проверяются на принадлежность корню checkout. Запрещены: абсолютные пути,
path traversal, symlink-escape (файл и каталог), чтение `.env`/секретов/`.git`. Чтение —
через `O_NOFOLLOW` + `fstat` (TOCTOU-safe), с лимитом размера. `is_safe_repo_path` валидирует
пути, предложенные моделью, до коммита.

## `planning`

Explore & Plan: разведка по песочнице → `finalize_plan` → `RiskPlanGate`. Если для
репозитория есть граф кода (graphify), `explore_and_plan` добавляет в промпт **навигационную
подсказку** (`graph_hint_untrusted`) — узлы и связи, релевантные бизнес-цели; это подсказка,
а не источник истины, и при её отсутствии планирование работает только на safe-tools. Пути
изменений от модели валидируются (`PlanError` на traversal/absolute/denied). `doc_impact`
fail-closed: без обоснования пропуска документация считается нужной.

## `coding`

**Агентный цикл с верификацией** (`generate_changes`): сгенерировать файлы по плану →
`self_check` (покрытие плана, нет лишних/пустых/небезопасных путей) → записать в checkout →
прогнать проверки репо (`lint`/`typecheck`/`test` из `.ai-agent.yml`) через `sandbox_exec` →
отдать падения модели → повторить до `MAX_CODE_ITERATIONS`. Результат несёт `verified`
(проверки прогнаны и зелёные) и попадает в описание MR.

Fail-closed: если после лимита итераций проверки не зелёные — MR не открывается (нужен
человек). Если проверок нет или изоляция не подтверждена — MR помечается **unverified** (его
смотрит человек), но не блокируется. Запрещено отключать тесты, менять thresholds или удалять
security-проверки.

Размер изменения дополнительно ограничен бюджетом scope (см. [Гейты](gates.md#scope)):
крупный дифф не получает авто-flip Draft→Ready и остаётся человеку.

`run_fix_loop` — тот же цикл для **самоисправления существующей ветки**: гоняет проверки репо
и чинит падения правками модели до зелёного либо до лимита (тогда fail-closed → человек).
Используется при упавшем CI и по команде `@ai fix` (оркестрация — `run_self_fix`,
см. [API и оркестрация](api-orchestration.md)).
