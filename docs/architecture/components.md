# Компоненты

Сервис разделён на публичный приём вебхуков, фоновые воркеры, хранилища состояния и
интеграции. Даже на одном хосте компоненты разнесены по контейнерам.

```
Bitrix24 / GitLab Webhooks
        │
        ▼
Reverse proxy / TLS / rate limit          (Caddy)
        │
        ▼
ai-developer-api  (FastAPI, быстрый ack)   ← main, webhooks
        │
        ▼
Task queue / Redis locks                   ← state, worker (Arq)
        │
        ▼
ai-developer-worker pool                   ← orchestrator + фазовые модули
  │         │          │           │
  ▼         ▼          ▼           ▼
PostgreSQL  Redis   GitLab API   LLM API
  │
  ▼
Audit log / metrics / alerts               ← audit, metrics
```

## Роли контейнеров

| Контейнер | Роль | Примечание |
|---|---|---|
| `api` | приём вебхуков и команд, валидация подписи, быстрый ack | CPU/RAM минимальные |
| `worker` | долгие фазы: брифинг, план, генерация, review, Red Team | масштабируется горизонтально; изолированный `/tmp`, cleanup |
| `postgres` | каноническое состояние брифинга, audit, task state | backup обязателен |
| `redis` | locks, идемпотентность, счётчики, бюджеты, снапшоты | только volatile |
| `proxy` | TLS, маршруты вебхуков, rate limit | публикует только вебхуки/health |

## Разделение источников истины

- **PostgreSQL** — источник истины для брифинга и состояния задач. Из истории комментариев
  состояние не восстанавливается.
- **Redis** — только volatile: блокировки, идемпотентность, счётчики правок и токенов,
  JSON-снапшоты. Авторитетные лимиты дублируются в БД, чтобы сброс Redis не обходил гейты.

GitLab CI/CD остаётся отдельным контуром: MR → GitLab Runner → статусы → вебхук → петля
самоисправления.
